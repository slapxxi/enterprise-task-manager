# Техническое задание: Enterprise Task Manager

- **Тип приложения**: Single Page Application (SPA).
- **Цель**: Разработать высокопроизводительный менеджер задач со сложной бизнес-логикой.

## 1. Технологический стек и ограничения

- [x] Язык: TypeScript (версия 5+).
  - [x] Обязательно: "strict": true в tsconfig.json.
  - [x] Запрещено: Использование типа any. Весь код должен быть строго типизирован.
- [x] Утилиты: Запрещены большие библиотеки вроде Lodash/Moment.js. Для работы с датами использовать нативные методы или легковесный dayJs.

## 2. Архитектура и Качество кода

- [x] Архитектурный паттерн: Приложение должно быть спроектировано по методологии FSD (Feature-Sliced Design) или Clean Architecture.
  - [x] Бизнес-логика должна быть отделена от UI-компонентов.
  - [x] Слои приложения должны иметь четкие границы ответственности.
- [x] State Management: Context

## 3. Работа с данными и Имитация API

Поскольку бэкенда нет, фронтенд должен вести себя так, будто он общается с ненадежным сервером.

- Mock API Layer: Весь доступ к данным должен проходить через сервисный слой, возвращающий Promise.
  - Реализовать искусственную задержку (delay) от 300мс до 1500мс на каждый запрос.
  - Реализовать Random Error Generator: каждый запрос имеет 15% шанс упасть с ошибкой (500 Internal Server Error).
- Optimistic UI: Интерфейс должен реагировать мгновенно.
  - При создании/удалении задачи UI обновляется сразу.
  - Если "сервер" вернул ошибку — изменение откатывается, пользователю показывается Toast-уведомление.
- Хранение (Persistence):
  - Данные хранить в IndexedDB (асинхронное локальное хранилище).
  - Реализовать паттерн "Repository" для абстракции работы с IndexedDB.

## 4. Функциональные требования (Core Logic)

### 4.1. Структура задачи

- [x] Задача — сложный объект:

- id: UUID.
- title: Строка (валидация: не пустая, макс 100 символов).
- description: Rich Text (поддержка переносов строк и ссылок).
- priority: Low, Medium, High (влияет на сортировку).
- deadline: Дата и Время.
- parentId: ID родительской задачи (для вложенности).
- status: 'queue' | 'development' | 'done' | 'expired'.

### 4.2. Древовидная структура (Recursive Tasks)

- [x] Поддержка бесконечной вложенности задач (Задачи могут иметь подзадачи).
- [x] Прогресс родительской задачи зависит от детей: если у задачи 4 подзадачи и 2 выполнены, прогресс родителя — 50%.
- [x] Родительскую задачу нельзя отметить выполненной вручную, если не выполнены все подзадачи.

### 4.3. Автоматическое устаревание (Background Workers)

- [x] Если наступает deadline, задача должна сменить статус на expired.
- [x] При смене статуса на expired задача должна визуально измениться (покраснеть) без перезагрузки страницы

## 5. UI/UX Требования

### 5.1. Производительность (Virtualization)

- [x] Приложение должно генерировать 10,000 тестовых задач (кнопка "Seed Data").
- [x] Список задач должен использовать Virtual Scrolling (рендерить только видимые элементы). Скролл должен оставаться плавным.

## 5.2. Drag-and-Drop (DnD)

- [x] Реализовать перетаскивание задач.
- [x] Reordering: Изменение порядка задач в списке.
- [x] Nesting: Перетаскивание задачи на другую задачу делает её подзадачей.

### 5.3. Фильтрация и Поиск

- [x] Поиск по подстроке (debounce 300ms) с подсветкой найденного текста внутри дерева задач.
- [x] Если подзадача найдена, родитель должен автоматически раскрыться.
